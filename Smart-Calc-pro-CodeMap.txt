  LINES 272–437 — MATH ENGINE (ME object)
────────────────────────────────────────────────────────────────────────────────
  A self-contained arithmetic engine. No eval(), no external library.
  Uses Dijkstra's Shunting-Yard algorithm to parse and evaluate expressions.

  ME.FNS (L280–294)
    Function lookup table. Maps string names → JS Math functions.
    Notable:
      sin/cos/tan        → degree versions (multiply input by π/180)
      sinr/cosr/tanr     → radian versions (used internally by graph engine)
      asin/acos/atan     → degree-output versions
      asinr/acosr/atanr  → radian-output versions
      fact               → factorial (integer only, max 170, >170 returns Infinity)
      mod                → modulo (returns NaN if b=0)

  ME.PREC (L295)  → operator precedence: + - = 1, * / = 2, ^ = 3
  ME.RASSO (L296) → right-associative set: only ^ is right-associative

  ME.tokenise(expr) (L297–333)
    Input: raw expression string
    Output: array of token objects {t, v}
      t: 'fn' | 'const' | 'num' | 'op'
      v: string name or numeric value

    Processing steps:
      1. Strip whitespace, normalise π→pi, ×→*, ÷→/
      2. Try match function name (longest-match first via regex order)
      3. Try match constant: pi, e, x  (uses (?![a-zA-Z0-9_]) lookahead
         instead of \b so it works after digits — e.g. "2x")
      4. Try match number (with optional decimal and scientific notation)
      5. Try match operator from set: + - * / ^ % ( )
      6. Unknown character → silently skip (L308)
      7. Post-process % tokens: postfix % (e.g. "8%") → inject /100 tokens
         binary % is left for the evaluator to flag as error

  ME.evaluate(expr, xVal) (L334–436)
    Input: expression string, optional x value (for graphing)
    Output: { result: number|null, error: string|null }

    Processing steps:
      1. Tokenise
      2. Implicit function parentheses:
           sin x → sin(x)
           sin2x → sin(2x)  [wraps next atom in parens if no ( follows fn]
      3. Implicit multiplication insertion:
           2x    → 2 * x
           3(x)  → 3 * (x)
           x(2)  → x * (2)
           Condition: left is value/const/closing paren
                      right is fn/const/num/opening paren
      4. Unary minus/plus handling:
           Leading - → insert 0 before it (0 - x evaluates correctly)
           Leading + → skip it (no-op)
      5. Shunting-Yard: tokens → RPN (Reverse Polish Notation) output queue
         Handles: operator precedence, right-associativity (^), function calls,
                  parentheses matching (returns error on mismatch)
      6. RPN evaluation: stack-based, applies operators/functions left to right
      7. Division by zero, non-finite result, unknown operator → error return

    x variable: when xVal is provided, 'x' tokens resolve to xVal.
                 when null, 'x' resolves to Math.E (design decision — prevents
                 errors in non-graph contexts where x appears by accident)

    graphFn       → current graph function string (e.g. "sin(x)")

────────────────────────────────────────────────────────────────────────────────
  memOp(op) handles: 'mc' | 'mr' | 'm+' | 'm-'

  mc → zero memory, set memSet=false, toast
  mr → insert memory value at cursor position in textarea.
       If textarea not present, sets _s.expr directly.
       Calls updateHighlight + updateDisplayOnly (no full re-render).
  m+ → add current result to memory
  m- → subtract current result from memory
  All ops call renderMemBar() to refresh the memory display strip.
  Error/invalid result states are checked before M+/M-.

────────────────────────────────────────────────────────────────────────────────
  LINES 776–1319 — GRAPH ENGINE CLASS (GraphEngine)
────────────────────────────────────────────────────────────────────────────────
  Full custom canvas graphing engine. No external library (e.g. no Desmos,
  no Chart.js). Renders directly to a <canvas> element.

  --- CONSTRUCTOR (L782–805) ---
    Parameters: canvas element, ME (math engine reference)
    Initialises:
      this.graphs      → Map of graph entries (id → {fn, type, color, segments})
      this.viewBox     → {xMin, xMax, yMin, yMax} (default -10 to +10 both axes)
      this.pixelRatio  → window.devicePixelRatio for HiDPI/Retina displays
      this.maxDepth    → 12 (max recursion depth for adaptive subdivision)
      this.colors[]    → palette for multiple graphs
      ResizeObserver   → calls resize() whenever canvas changes size
    Calls: resize(), setupInteractions()

  --- COORDINATE HELPERS (L807–842) ---
    _niceStep(range, targetTicks)
      Calculates a "nice" grid step: prefers 1, 2, 5, 10 multiples.
      Used for grid lines and axis labels.

    _colors()
      Returns theme-aware color object (bg, grid, axis, label, origin).
      Reads _s.theme — dark/light aware.

    xToPx(x, w) → converts math x-coordinate to canvas pixel x
    yToPy(y, h) → converts math y-coordinate to canvas pixel y (inverted)
    pxToX(px, w) → inverse: canvas pixel → math x
    pyToY(py, h) → inverse: canvas pixel → math y

  --- RESIZE (L844–853) ---
    resize()
      Reads canvas CSS size via getBoundingClientRect().
      Sets canvas.width/height to CSS size × pixelRatio (HiDPI sharpness).
      Resets transform, applies pixelRatio scale.
      Calls invalidate() to trigger redraw.

  --- GRAPH MANAGEMENT (L855–873) ---
    setGraph(fn, type, opt)
      Adds or replaces a graph entry in this.graphs Map.
      type: 'explicit' | 'polar' | 'vertical'
      opt: { id, color }
      Calls invalidate() → triggers async render.

    clearGraphs() → empties the map, invalidates.

  --- SAMPLING / CURVE COMPUTATION (L875–1007) ---

    _toRadFn(expr) (L881–893)
      Converts degree trig names to radian variants before evaluation.
      Order matters: asin before sin (prevents partial match).
      sin→sinr, cos→cosr, tan→tanr, asin→asinr, acos→acosr, atan→atanr
      sinr/cosr/tanr already radians → (?!r) lookahead skips them.
      Only used by the graph engine — calculator tabs call ME.evaluate directly
      and use degree versions intentionally.

    evalSafe(fn, x) (L895–901)
      Calls _toRadFn then ME.evaluate(fn, x).
      Returns null on any error, NaN, Infinity, or null result.
      Null signals a domain gap (e.g. log of negative, 1/0).

    _toPx(x, y, w, h) (L903–910)
      Converts a math-space point to [pixelX, pixelY].
      Used internally by subdivide for error measurement.

    subdivide(fn, x1, y1, x2, y2, depth, w, h) (L912–945)
      Recursive adaptive subdivision for curve smoothness.
      Algorithm:
        1. Compute midpoint: midX = (x1+x2)/2, midY = evalSafe(midX)
        2. If midY is null → domain gap → return null
        3. Convert all three points to pixel space
        4. Measure pixel distance from actual mid to straight-line mid
        5. If error < 0.4px → flat enough → return endpoints only
        6. Otherwise → recurse left half + right half
        7. Stop at maxDepth=12 (prevents infinite recursion on discontinuities)
      Returns array of {x,y} points in math space, or null on domain gap.

    computeExplicit(fn) (L947–996)
      Main curve sampler for y=f(x) type graphs.
      Steps:
        1. Determine base sample count: max(100, canvasWidth/2) intervals
        2. For each interval [x1, x2]:
           a. Evaluate y1, y2 at endpoints
           b. If either is null → flush current segment, continue
           c. Run subdivide → get refined point array
           d. If subdivide returns null → flush, continue
           e. Discontinuity check: measure pixel jump from last accepted point
              to new batch. If vertical jump > 8% of canvas height AND
              steeper than 45° → it's an asymptote → flush and start new segment
           f. Append points to current segment (deduplicating shared endpoint)
        3. flush() collects complete segments (≥2 points)
      Returns: array of segments, each segment is array of {x,y} math points.
      Each segment is drawn as a continuous line. Gaps between segments
      represent domain gaps or asymptotes.

    computePolar(fn) (L999–1007)
      Simple polar sampler: t from 0 to 2π, step 0.01.
      Converts (r, t) → (x, y) Cartesian.
      Returns single segment (no gap detection).

  --- UPDATE DISPATCH (L1009–1025) ---
    updateGraphs()
      Called before every render frame.
      Iterates this.graphs, recomputes segments only if _needsUpdate=true.
      Dispatches by type:
        'vertical' → builds a 2-point segment: x=xVal from yMin-1 to yMax+1
                     (the extra ±1 ensures it extends through the full viewport
                     even at edge of viewBox). fn stores the xVal as a string.
        'explicit' → computeExplicit(g.fn)
        'polar'    → computePolar(g.fn)
      Sets _needsUpdate=false after completion.

  --- AUTO Y-RANGE (L1027–1051) ---
    autoYRange(fn)
      Samples fn at 121 evenly-spaced x points across current xMin→xMax.
      Sorts valid y values, trims 5% outliers from each tail (handles spikes).
      Sets viewBox yMin/yMax with 15% padding.
      Falls back to ±10 if fewer than 3 valid samples.
      Called on initial plot and on double-click reset.
      NOT called if user has panned/zoomed Y axis (_userZoomedY flag, though
      note: _userZoomedY is currently not set anywhere — Y always auto-ranges
      unless the user pans, at which point it stays where they left it because
      drawGraph only calls autoYRange when !_ge._userZoomedY).

  --- RENDER LOOP (L1053–1095) ---
    invalidate()
      Sets _needsUpdate=true, calls requestRender().

    requestRender()
      Queues a single rAF (requestAnimationFrame).
      The _renderPending flag prevents duplicate rAF queues.
      On next frame: calls updateGraphs() then render().

    render()
      Master draw sequence per frame:
        1. Fill background
        2. drawGrid  → grid lines
        3. drawAxes  → x/y axis lines + origin dot
        4. drawAxisLabels → numeric tick labels
        5. ctx.clip to canvas bounds
        6. For each visible graph: drawGraph(g)

  LINES 1324–1394 — EQUATION SANITIZER  [KEY FUNCTION — core fix]
────────────────────────────────────────────────────────────────────────────────
  sanitizeGraphExpr(raw)

  Purpose:
    Normalises any user equation input into a form the graph engine understands.
    Returns an object: { type, fn } or { type, xVal, label }

  Return types:
    { type: 'explicit', fn: '...' }
      → Normal curve. fn is a clean expression of x. Graph plots y = fn.

    { type: 'vertical', xVal: number, label: 'x = N' }
      → Vertical line. Graph draws a vertical line at x = xVal.

  Decision tree (in order):
    1. No '=' in input
       → { type:'explicit', fn: raw }
       Example: "sin(x)" → explicit sin(x)

    2. '=' present → split at first '=' into lhs and rhs

    3. y = f(x)  (lhs is "y")
       → { type:'explicit', fn: rhs }
       Example: "y = 2x+5" → explicit 2x+5

    4. f(x) = y  (rhs is "y")
       → { type:'explicit', fn: lhs }
       Example: "2x+5 = y" → explicit 2x+5

    5. x = constant  (lhs is "x")
       → evaluates rhs as a constant
       → { type:'vertical', xVal: N, label: 'x = N' }
       Example: "x = 3" → vertical at 3
       Example: "x = -2.5" → vertical at -2.5

    6. f(x) = g(x)  (rhs contains x)
       → { type:'explicit', fn: '(lhs)-(rhs)' }
       Plots the difference curve. Zero-crossings = solutions.
       Example: "2x = x+1" → explicit (2x)-(x+1) → plots y = x-1

    7. f(x) = constant  (rhs has no x)
       a. Evaluate rhs → constant C
       b. Build shiftedFn = "(lhs)-(C)"
       c. Sample shiftedFn at x=0, x=1, x=2 → get f0, f1, f2
       d. Compute second difference: |(f2-f1) - (f1-f0)|
       e. If second difference < 1e-9 → expression is LINEAR
          → slope = f1 - f0
          → xSol = -f0 / slope  (exact analytical solution)
          → { type:'vertical', xVal: xSol, label: 'x = N' }
          Example: "2x+5 = 0" → f0=5, slope=2 → xSol=-2.5 → vertical at -2.5
          Example: "5x-10 = 0" → f0=-10, slope=5 → xSol=2 → vertical at 2
       f. If not linear (quadratic, trig, etc.) OR slope≈0
          → { type:'explicit', fn: shiftedFn }
          Plots shifted curve. User sees zero-crossings = solutions.
          Example: "x^2-4 = 0" → explicit (x^2-4)-(0) → parabola, roots at ±2

    8. Fallback (lhs has no x, or parse failure)
       → { type:'explicit', fn: shiftedFn }

  ARCHITECTURE NOTES FOR FUTURE YOU
================================================================================

  1. ALL graph input goes through sanitizeGraphExpr() first.
     Never call ME.evaluate or _ge.setGraph with raw _s.graphFn.

  2. GraphEngine is stateful. _ge persists across plot calls.
     It is only recreated when the canvas DOM element changes
     (which happens on tab switch because renderMain() replaces innerHTML).

  3. The 30ms setTimeout in renderGraph() is load-bearing.
     The canvas needs to be in the DOM and laid out before GraphEngine
     can read its size. Removing it causes 0×0 canvas bugs.

  4. render() = full re-render. Expensive. Only call from:
     - Mode switches (set-mode)
     - computeExpr (= button)
     - Clear (ac action)
     - Tab-level state changes (eq-type, fml-cat, etc.)
     For everything in the calculator: use updateDisplayOnly() + renderAC().

  5. The math engine uses degrees for sin/cos/tan in the calculator.
     The graph engine converts to radians via _toRadFn() before sampling.
     This is intentional — the two contexts have different conventions.
     Do not change this without updating both the ME.FNS table and _toRadFn.

  6. sanitizeGraphExpr linearity detection works by sampling at x=0,1,2.
     It assumes the expression is defined at all three points.
     Edge cases where f(0) or f(1) is undefined (e.g. 1/x = 0) fall through
     to the 'explicit' branch and plot the curve instead. This is safe.

  7. History is capped at 10 display entries, 50 keyboard nav entries.
     Both are arrays in _s, history also persisted to localStorage as 'sc3_hist'.
